# Walmart Sales Forecasting

Kaggle-ის [Walmart Recruiting - Store Sales Forecasting](https://www.kaggle.com/competitions/walmart-recruiting-store-sales-forecasting) კონკურსის ამოხსნა. ამოცანაა 45 Walmart-ის მაღაზიის დეპარტამენტების კვირეული გაყიდვების პროგნოზირება. შეფასების მეტრიკა არის WMAE (weighted mean absolute error), სადაც სადღესასწაულო კვირებს 5-ჯერ მეტი წონა აქვს — ამიტომ Super Bowl, Labor Day, Thanksgiving და Christmas კვირების სწორად პროგნოზირება განსაკუთრებით მნიშვნელოვანია.

## მონაცემები

- **train.csv** — 421,570 ჩანაწერი: 45 მაღაზია, 81 დეპარტამენტი, 2010-02-05 – 2012-10-26
- **test.csv** — 115,064 ჩანაწერი: 2012-11-02 – 2013-07-26
- **features.csv** — Temperature, Fuel_Price, CPI, Unemployment, MarkDown1-5
- **stores.csv** — მაღაზიის ტიპი (A/B/C) და ზომა

## EDA (eda.ipynb)

პირველ ეტაპზე მონაცემები დეტალურად გავიარეთ, რამაც რამდენიმე მნიშვნელოვანი რამ გამოავლინა:

- **MarkDown სვეტები თითქმის ცარიელია** — 2010 წელს 100%, 2011 წელს ~85% missing. რეალურად მხოლოდ 2012 წლიდან ივსება. ამიტომ მათზე დიდი იმედის დამყარება არ შეიძლება, NA-ები 0-ით შევავსეთ (მარკდაუნის არარსებობა ფაქტობრივად ნიშნავს რომ აქცია არ იყო).
- **Weekly_Sales ძლიერ skewed არის** — მედიანა 7,612, საშუალო 15,981, მაქსიმუმი 693,099. 1,285 ჩანაწერი უარყოფითია (სავარაუდოდ დაბრუნებები), ისინი 0-ზე დავჭერით.
- **Christmas-ის IsHoliday ალამი არასწორ კვირაზეა** — ეს ყველაზე საინტერესო აღმოჩენა იყო. დატასეტში სადღესასწაულოდ 31 დეკემბრის კვირაა მონიშნული, მაგრამ რეალური პიკი წინა კვირაშია: 2010-12-24 კვირის ჯამური გაყიდვები 80.9 მლნ იყო, ხოლო "ოფიციალური" სადღესასწაულო 2010-12-31 კვირის — მხოლოდ 40.4 მლნ. ამის გამო ყველა მოდელს დავამატეთ ცალკე `IsPreChristmas` ფიჩერი.
- **დეპარტამენტები ძალიან განსხვავდება** — Dept 92-ის საშუალო 75,205-ია, ბოლო დეპარტამენტების კი თითქმის 0. Store/Dept იდენტიფიკატორები ყველაზე ძლიერი სიგნალია.
- **გარე ფაქტორები სუსტია** — Temperature, Fuel_Price, CPI, Unemployment ცალ-ცალკე თითქმის არ კორელირებს გაყიდვებთან.
- **test-ში 11 (Store, Dept) წყვილი საერთოდ არ არსებობს train-ში** — ამან თითო-სერიაზე დამოკიდებული მოდელებისთვის (Prophet, DLinear) fallback ლოგიკის დამატება მოითხოვა.

## ვალიდაცია

რადგან ამოცანა time-series-ია, ჩვეულებრივი random K-Fold არ გამოდგება — მომავლის მონაცემები წარსულში "გაჟონავდა". ამიტომ ყველა მოდელი ერთნაირი **დროზე დაფუძნებული 3-fold სქემით** ვაფასებთ: ყოველ fold-ში ვსწავლობთ მხოლოდ წარსულზე და ვაფასებთ მომდევნო ~8 თვეზე. მეტრიკა ყველგან WMAE-ა, იგივე წონებით რაც kaggle-ზე.

ერთი დეტალი: პირველი fold-ის სატრენინგო პერიოდი მხოლოდ ~35 კვირაა, რაც DLinear-ის 52-კვირიან input window-ზე ნაკლებია — ამან თავიდან crash გამოიწვია და მოდელს fallback დავუმატეთ (window ავტომატურად პატარავდება, თუ ისტორია არ ჰყოფნის).

## მოდელები

ყველა მოდელი შენახულია sklearn Pipeline-ად, რომელიც პირდაპირ raw მონაცემებზე ეშვება (preprocessing pipeline-შივეა ჩაშენებული).

### XGBoost (model_experiment_XGBoost.ipynb)

ფიჩერები: Store, Dept, Size, Type, IsHoliday, თარიღის კომპონენტები (Year, Month, Week), IsPreChristmas, MarkDown1-5, ეკონომიკური ინდიკატორები. Feature importance-მა აჩვენა რომ ყველაზე მნიშვნელოვანია Dept (0.25) და Size (0.22), საინტერესოა რომ ჩვენი დამატებული IsPreChristmas მე-6 ადგილზეა (0.06) — ანუ Christmas-ის კვირის კორექციამ რეალურად იმუშავა.

შედეგები (n_estimators=500, max_depth=8, lr=0.05):

| Fold | WMAE |
|------|------|
| 1 | 4116.5 |
| 2 | 2950.4 |
| 3 | 2555.8 |
| **საშუალო** | **3207.6** |

პირველი fold-ის შედარებით ცუდი შედეგი ლოგიკურია — ტრენინგში ჯერ სრული წელიც არ იყო და მოდელს სეზონურობის დანახვა არ შეეძლო. Full-train WMAE 2140-ია.

### LightGBM (model_experiment_LightGBM.ipynb)

იგივე ფიჩერები რაც XGBoost-ს, რომ შედეგები პირდაპირ შედარებადი იყოს. განსხვავება ჰიპერპარამეტრებშია — LightGBM leaf-wise იზრდება და მისთვის num_leaves (127) უფრო მნიშვნელოვანი პარამეტრია ვიდრე max_depth. ტრენინგი საგრძნობლად სწრაფია.

### Prophet (model_experiment_Prophet.ipynb)

კლასიკური სტატისტიკური მიდგომა. თითო store-dept სერიაზე ცალკე Prophet-ის მორგება (3,331 მოდელი) ძალიან ნელი იქნებოდა, ამიტომ top-down მიდგომა ავირჩიეთ:

1. თითო მაღაზიის ჯამურ გაყიდვებზე ვარგებთ Prophet-ს (სულ 45 მოდელი) — yearly seasonality + კონკურსის 4 დღესასწაული და pre-Christmas კვირა holiday რეგრესორებად.
2. მაღაზიის პროგნოზს დეპარტამენტებზე ვანაწილებთ ისტორიული წილებით — თითო დეპარტამენტის წილი კვირის ნომრის (week-of-year) მიხედვით, რომ სეზონური დეპარტამენტები (მაგ. სააღდგომო/საშობაო საქონელი) სწორად დაიყოს.

შედეგები:

| Fold | WMAE |
|------|------|
| 1 | 54362.5 |
| 2 | 1961.1 |
| 3 | 1756.4 |

პირველი fold-ის შედეგი კატასტროფულია და ეს თავად Prophet-ის ბუნებიდან გამომდინარეობს: ტრენინგში მხოლოდ ~35 კვირა იყო — სრულ წელზე ნაკლები — და yearly seasonality-ის სწავლა შეუძლებელი იყო, ექსტრაპოლაცია კი Thanksgiving/Christmas პერიოდზე მოხვდა. სამაგიეროდ იმ fold-ებზე, სადაც სრული წელი ჰქონდა ნანახი, Prophet-მა ტრი-მოდელებზე უკეთესი შედეგიც კი აჩვენა (1961 და 1756 vs XGBoost-ის 2950 და 2556). დასკვნა: კლასიკურ სტატისტიკურ მოდელებს მინიმუმ ერთი სრული სეზონური ციკლი სჭირდებათ, თუმცა საკმარისი ისტორიის პირობებში ძალიან კონკურენტულია.

### DLinear (model_experiment_DLinear.ipynb)

Deep learning მოდელი [DLinear სტატიიდან](https://arxiv.org/abs/2205.13504) — იდეა ისაა, რომ სერია moving average-ით იშლება trend და seasonal კომპონენტებად და თითოეულზე ცალკე linear layer მუშაობს. მარტივი არქიტექტურაა, მაგრამ სტატიაში ტრანსფორმერებზე უკეთესი შედეგები აქვს long-term forecasting-ში.

ჩვენი იმპლემენტაცია: 52-კვირიანი window → შემდეგი კვირის პროგნოზი. თითო სერია ნორმალიზდება საკუთარი mean/std-ით (სხვანაირად დიდი მაღაზიები დომინირებდნენ loss-ში). Loss-ად პირდაპირ weighted MAE გამოვიყენეთ, იგივე 5x წონებით სადღესასწაულო კვირებზე — ანუ მოდელი პირდაპირ იმ მეტრიკაზე ოპტიმიზდება, რომლითაც ფასდება. test-ის პერიოდისთვის პროგნოზი autoregressive-ად ეშვება (ყოველი ნაპროგნოზები კვირა შემდეგი კვირის window-ში ხვდება).
