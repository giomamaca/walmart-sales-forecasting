# Walmart Sales Forecasting
## მონაცემები

- train.csv: 421,570 ჩანაწერი. 45 მაღაზია, 81 დეპარტამენტი, 2010-02-05-დან 2012-10-26-მდე
- test.csv: 115,064 ჩანაწერი, 2012-11-02-დან 2013-07-26-მდე

## EDA (eda.ipynb)

პირველ ეტაპზე მონაცემები დეტალურად შევისწავლეთ და რამდენიმე მნიშვნელოვანი რამ გამოჩნდა.

MarkDown სვეტები თითქმის ცარიელია: 2010 წელს 100%, 2011 წელს დაახლოებით 85% გამოტოვებულია, და რეალურად მხოლოდ 2012 წლიდან იწყებს შევსებას. ამიტომ მათზე ძლიერ დაყრდნობა არ შეიძლება. NA-ები 0-ით შევავსეთ, რადგან მარკდაუნის არარსებობა, პრაქტიკულად, იმას ნიშნავს, რომ აქცია არ ჩატარებულა.

Weekly_Sales-ის განაწილება ძლიერ ასიმეტრიულია. მედიანა 7,612-ია, საშუალო 15,981, მაქსიმუმი 693,099. 1,285 ჩანაწერი უარყოფითია (სავარაუდოდ დაბრუნებები), ამიტომ ისინი 0-ით ჩავანაცვლეთ.

ყველაზე საინტერესო აღმოჩენა ის იყო, რომ Christmas-ისთვის განკუთვნილი IsHoliday ინდიკატორი არასწორ კვირაზეა. მონაცემთა ნაკრებში სადღესასწაულოდ 31 დეკემბრის კვირაა მონიშნული, მაგრამ რეალური პიკი წინა კვირაშია: 2010-12-24-ის კვირის ჯამური გაყიდვები 80.9 მლნ იყო, "ოფიციალური" სადღესასწაულო 2010-12-31-ის კვირის კი მხოლოდ 40.4 მლნ. ამის გამო ყველა მოდელს დავამატეთ ცალკე IsPreChristmas ნიშანი. ეს დაკვირვება მოგვიანებით კიდევ უფრო მნიშვნელოვანი აღმოჩნდა, როცა კალენდარულად გასწორებულ lag-ებზე გადავედით (იხ. ქვემოთ).

დეპარტამენტები ერთმანეთისგან ძალიან განსხვავდება. Dept 92-ის საშუალო გაყიდვები 75,205-ია, ბოლო დეპარტამენტებში კი ეს მაჩვენებელი თითქმის 0-ს უტოლდება. ყველაზე ძლიერ სიგნალს Store-ისა და Dept-ის იდენტიფიკატორები იძლევა.

გარე ფაქტორები სუსტია. Temperature, Fuel_Price, CPI და Unemployment ცალ-ცალკე გაყიდვებთან თითქმის არ კორელირებს.

test მონაცემებში არსებული 11 (Store, Dept) წყვილი train მონაცემებში საერთოდ არ გვხვდება. ამან თითო სერიაზე დამოკიდებული მოდელებისთვის (Prophet, DLinear) fallback ლოგიკის დამატება მოითხოვა.

## ვალიდაცია

რადგან ამოცანა დროით მწკრივებს ეხება, ჩვეულებრივი random K-Fold არ გამოდგება, რადგან ამ შემთხვევაში მომავლის მონაცემები წარსულში „გაჟონავდა“. ამიტომ ყველა მოდელს ერთნაირი, დროზე დაფუძნებული 3-fold სქემით ვაფასებთ: თითოეულ fold-ში მოდელს მხოლოდ წარსულ მონაცემებზე ვასწავლით და მომდევნო დაახლოებით რვა თვეზე ვაფასებთ. მეტრიკა ყველგან WMAE-ა, იმავე წონებით, რომლებიც Kaggle-ზე გამოიყენება.

გასათვალისწინებელია ერთი დეტალიც: პირველი fold-ის სატრენინგო პერიოდი მხოლოდ 35 კვირაა, რაც DLinear-ის 52-კვირიან input window-ზე ნაკლებია. ამან თავდაპირველად შეცდომა გამოიწვია და მოდელს fallback დავუმატეთ (window ავტომატურად პატარავდება, თუ ისტორიული მონაცემები საკმარისი არ არის).

პროექტის შუა ეტაპზე უფრო სერიოზული პრობლემაც აღმოვაჩინეთ. ლოკალურად მოდელები კარგ შედეგებს (WMAE 2000-2700) გვაძლევდნენ, Kaggle-ზე კი 3200-3800 გვიჩვენებდა, და ეს სხვაობა ყველა მოდელს დაახლოებით თანაბრად ჰქონდა. ეს იმაზე მიანიშნებდა, რომ პრობლემა კონკრეტულ მოდელში კი არა, თვითონ ვალიდაციაში იყო. ცალკე დიაგნოსტიკური notebook-ით (diagnostic.ipynb) გავარკვიეთ მიზეზი: მესამე fold-ის სავალიდაციო პერიოდში (2012 წლის თებერვლიდან ოქტომბრამდე) Thanksgiving და Christmas საერთოდ არ ხვდება, ანუ ის ხელოვნურად „მარტივი“ იყო და საშუალო შეფასებას აუმჯობესებდა. Kaggle-ის ტესტი კი სწორედ ნოემბრიდან იწყება, ყველაზე რთული სადღესასწაულო პიკებით.

ამის გამო მომდევნო მოდელებში დავამატეთ ცალკე, ტესტის ფორმის holdout: მოდელს ვასწავლით 2011 წლის ოქტომბრამდე და ვაფასებთ 2011 ნოემბრიდან 2012 ივლისამდე. ეს ფანჯარა კალენდარულად ზუსტად ისეთია, როგორიც Kaggle-ის ტესტი (ნოემბრიდან ივლისამდე, სადღესასწაულო პიკებით), და დიაგნოსტიკამ დაადასტურა, რომ Kaggle-ის ქცევას ის ბევრად უკეთ იმეორებს, ვიდრე 3-fold CV. ამიტომ გვიანდელი მოდელების ჰიპერპარამეტრებს სწორედ ამ holdout-ზე ვარჩევდით, 3-fold CV კი ყველა მოდელს მაინც ვუთვალეთ, რომ ძველ და ახალ შედეგებს შორის შედარებადობა შენარჩუნებულიყო.

## მოდელები

ყველა მოდელი შენახულია sklearn Pipeline-ად, რომელიც პირდაპირ დაუმუშავებელ მონაცემებზე მუშაობს (preprocessing pipeline-შივეა ჩაშენებული). ჯერ თითო არქიტექტურის საბაზისო ვერსია ავაწყვეთ, შემდეგ კი ეტაპობრივად ვაუმჯობესებდით: ჰიპერპარამეტრების ძებნა, ისტორიული ნიშნები, leakage-ის გამოსწორება. ძველი ვერსიები შედარებისთვის უცვლელად შევინახეთ.

### XGBoost (model_experiment_XGBoost.ipynb)

ნიშნები: Store, Dept, Size, Type, IsHoliday, თარიღის კომპონენტები (Year, Month, Week), IsPreChristmas, MarkDown1-5, ეკონომიკური ინდიკატორები. Feature importance-მა აჩვენა, რომ ყველაზე მნიშვნელოვანია Dept (0.25) და Size (0.22). საინტერესოა, რომ ჩვენი დამატებული IsPreChristmas მე-6 ადგილზეა (0.06), ანუ Christmas-ის კვირის კორექციამ რეალურად იმუშავა.

შედეგები (n_estimators=500, max_depth=8, lr=0.05):

| Fold | WMAE |
|------|------|
| 1 | 4116.5 |
| 2 | 2950.4 |
| 3 | 2555.8 |
| საშუალო | 3207.6 |

პირველ fold-ში მიღებული შედარებით ცუდი შედეგი მოსალოდნელი იყო. სასწავლო მონაცემებში ჯერ სრული წელიც არ იყო წარმოდგენილი, ამიტომ მოდელი სეზონურობას სათანადოდ ვერ სწავლობდა. Full-train WMAE 2140-ია.

### LightGBM (model_experiment_LightGBM.ipynb)

იგივე ნიშნები გამოვიყენეთ, რაც XGBoost-ში, რათა შედეგები პირდაპირ შეგვედარებინა. განსხვავება ჰიპერპარამეტრებშია: LightGBM leaf-wise იზრდება და მისთვის num_leaves (127) უფრო მნიშვნელოვანი პარამეტრია, ვიდრე max_depth. ამასთან, ტრენინგიც საგრძნობლად სწრაფია.

შედეგები (n_estimators=1000, num_leaves=127, lr=0.05):

| Fold | WMAE |
|------|------|
| 1 | 3988.6 |
| 2 | 2845.2 |
| 3 | 2265.8 |
| საშუალო | 3033.2 |

სამივე fold-ზე XGBoost-ზე უკეთესია (საშუალო 3033 vs 3208). Full-train WMAE 1682-ია, ხეზე დაფუძნებულ მოდელებს შორის ყველაზე დაბალი. leaf-wise ზრდა მოდელს მეტ მოქნილობას აძლევს და CV-ზე ოვერფიტის ნიშნები არ ჩანს.

### ჰიპერპარამეტრების ძებნა (model_experiment_XGBoost_v2.ipynb, model_experiment_LightGBM_v2.ipynb, model_experiment_Prophet_v2.ipynb, model_experiment_DLinear_v2.ipynb)

პირველ ვერსიებში ჰიპერპარამეტრები ხელით გვქონდა არჩეული, ამიტომ ყველა არქიტექტურას random search დავუმატეთ: თითო trial-ში კონფიგურაცია საძებნი სივრციდან სემპლდება, ფასდება იმავე დროზე დაფუძნებული ვალიდაციით და ყველა trial-ი MLflow/Wandb-ზე ილოგება.

შედეგმა ხეზე დაფუძნებული მოდელები შესამჩნევად გააუმჯობესა: XGBoost-ის საშუალო CV 3208-დან 2884-მდე დაეცა, LightGBM-ის კი 3033-დან 2918-მდე. Prophet-ში ტიუნინგმა ყველაზე თვალსაჩინო ეფექტი პირველ fold-ზე მოახდინა: ძლიერმა seasonality prior-მა (seasonality_prior_scale=0.1) მოდელს 35 კვირიდან წლიური ციკლის „გამოგონება“ აუკრძალა და პირველი fold-ის შეცდომა 54362-დან 4114-მდე შემცირდა, დანარჩენ fold-ებზე კი შედეგი იგივე დარჩა (1961 და 1757). DLinear-ის შემთხვევაში კი პატარა საძებნი ბიუჯეტი (8 trial-ი) საკმარისი არ აღმოჩნდა და ნაპოვნმა კონფიგურაციამ საბაზისო ვერსიას ვერ აჯობა (3171 vs 3017). ესეც სასარგებლო გაკვეთილი იყო: მცირე ბიუჯეტის random search-ს შემთხვევითობა მნიშვნელოვნად მართავს.

### lag ნიშნები და leakage (model_experiment_LightGBM_v3.ipynb, model_experiment_LightGBM_v4.ipynb)

აქამდე ხეზე დაფუძნებული მოდელები წარსულ გაყიდვებს საერთოდ ვერ ხედავდნენ, მხოლოდ კალენდარსა და მაღაზიის აღწერილობას. v3-ში ისტორიული ნიშნები დავამატეთ: Sales_Lag52 (გასული წლის იგივე კვირა), თითო სერიის საშუალო/მედიანა/სტანდარტული გადახრა, week-of-year საშუალო, დეპარტამენტისა და მაღაზიის საშუალოები. v4-ში ამას კალენდარულად გასწორებული lag-ებიც დაემატა (იხ. ქვემოთ).

აქ პროექტის ყველაზე საინტერესო ეპიზოდი მოხდა. ლოკალურად ორივე ვერსია აქამდე ნანახზე ბევრად უკეთეს შედეგებს გვაძლევდა, მაგრამ Kaggle-ზე პირიქით, თანდათან უარესდებოდა: 3200-დან ჯერ 3400-მდე, შემდეგ 3700-მდე. რაც უფრო მეტ ისტორიულ ნიშანს ვამატებდით, ლოკალური შედეგი უმჯობესდებოდა და Kaggle-ის შედეგი უარესდებოდა. ეს target leakage-ის კლასიკური სურათია.

პრობლემა აგრეგატულ ნიშნებში აღმოჩნდა (Series_Mean, Series_WoyMean და მსგავსები). ისინი fit-ის დროს ტრენინგის ყველა სტრიქონზე ითვლებოდა, თვითონ იმ სტრიქონის ჩათვლით. week-of-year საშუალოში, სადაც სერიას მოცემული კვირა ისტორიაში მხოლოდ 2-3-ჯერ აქვს, სტრიქონის საკუთარი გაყიდვები ამ ნიშნის მნიშვნელობის 33-50%-ს შეადგენდა. მოდელი ფაქტობრივად სწავლობდა პასუხის ნაწილობრივ გადაწერას საკუთარი ნიშნიდან; ტესტზე, სადაც სტრიქონი ისტორიაში არ არის, ეს „დამხმარე“ ქრებოდა და პროგნოზი ინგრეოდა.

### LightGBM_v5: leave-one-out (model_experiment_LightGBM_v5.ipynb)

გამოსავალი leave-one-out მიდგომა იყო: თუ სტრიქონი fit-ის მონაცემებში მონაწილეობდა, აგრეგატს მის საკუთარ წვლილს ვაკლებთ ((sum - own) / (n - 1)); ტესტის სტრიქონები კი, რომლებიც fit-ში არ ყოფილა, სრულ აგრეგატს იღებენ. ასე ტრენინგისა და ტესტის ნიშნებს ზუსტად ერთი და იგივე სემანტიკა აქვთ და აგრეგატში სტრიქონის საკუთარი პასუხი ვეღარასდროს ხვდება. ამასთან, ჰიპერპარამეტრებს უკვე ორ ტესტის ფორმის holdout-ზე (2010-2011 და 2011-2012) ვარჩევდით, რომ ერთი კონკრეტული ფანჯრის შემთხვევითობას არ მოვრგებოდით.

LightGBM_v5 ჩვენი საბოლოო, საუკეთესო მოდელია: holdout WMAE 1936, 3-fold CV საშუალო 2721, Kaggle-ზე კი დაახლოებით 2800. leakage-ის გასწორებამ ლოკალური შეფასება უფრო „მკაცრი“ გახადა, სამაგიეროდ Kaggle-ის შედეგი 3700-დან 2800-მდე დაწია, ანუ რეალური განზოგადება მკვეთრად გაუმჯობესდა.

### კალენდარულად გასწორებული lag-ები

ეს იდეა ცალკე აღწერას იმსახურებს, რადგან პირდაპირ EDA-ს მთავარ აღმოჩენას ეყრდნობა. ჩვეულებრივი Sales_Lag52 ზუსტად 52 კვირას (364 დღეს) იხედება უკან, ანუ გასული წლის იმავე ნომრის კვირას უყურებს. მაგრამ Christmas ყოველ წელს კვირის სხვა დღეზე მოდის: 2010 და 2011 წლებში მთავარი სავაჭრო დღეები 24/23 დეკემბრის კვირაში ჯდებოდა, 2012-ში (ტესტის წელს) კი კვირის საზღვრების წანაცვლების გამო ნაწილი შემდეგ კვირაში გადავიდა. კვირის ნომრით გასწორებული lag ამ პიკს ერთი კვირით აცდენს, WMAE-ს 5-მაგი წონა კი ამ შეცდომას განსაკუთრებით მკაცრად სჯის.

გამოსავალი: Sales_CalLag1y სამიზნე კვირის შვიდივე დღეს ცალ-ცალკე უყურებს, თითოეულისთვის პოულობს ზუსტად ერთი წლის წინანდელ იმავე კალენდარულ დღეს, ადგენს, რომელ Walmart-კვირაში იჯდა ის დღე, და გასული წლის კვირების მნიშვნელობებს დღეების რაოდენობის პროპორციული წონებით ურევს. ასე პიკის კალენდარული წანაცვლება თვითონ ნიშანშივეა ჩაშენებული და მოდელს ის მონაცემებიდან სწავლა აღარ სჭირდება (რასაც ვერც შეძლებდა, რადგან ტრენინგში ორივე Christmas ერთნაირად იყო განლაგებული). მაგალითად, 2012-12-28-ით დამთავრებული სატესტო კვირისთვის ჩვეულებრივი lag მთლიანად აცდენს გასული წლის პიკს, კალენდარული lag კი მის 2/7-ს სწორ კვირაში გადმოიტანს.

### ARIMA / SARIMA

კლასიკური ოჯახის საბაზისო მოდელებია, თეორიულად განვიხილეთ, თუმცა ტრენინგზე დრო არ დაგვიხარჯავს. ARIMA(p,d,q) სამი კომპონენტისგან შედგება: AR(p) ნიშნავს, რომ მიმდინარე მნიშვნელობა წარსული p მნიშვნელობის წრფივი კომბინაციაა, MA(q) — წარსული q პროგნოზის შეცდომების, ხოლო I(d) იმას ნიშნავს, რომ სერია d-ჯერ სხვაობდება (differencing) რათა ტრენდი მოშორდეს და სტაციონარული გახდეს. SARIMA ამას სეზონურ (P,D,Q,s) კომპონენტს უმატებს, ჩვენს კვირეულ მონაცემებზე s=52 იქნებოდა.

პრაქტიკაში ვერ გამოვიყენებდით რამდენიმე მიზეზით: 3,331 სერიაზე თითოეულისთვის order-ის შერჩევა პრაქტიკულად შეუძლებელია; s=52 სეზონური პერიოდი SARIMA-სთვის გამოთვლითად ძალიან მძიმეა, და მრავალი დღესასწაულის ეფექტის ჩამატება Prophet-ში ბევრად მარტივია. ამიტომ კლასიკური ოჯახიდან Prophet ავირჩიეთ, რომელიც იმავე იდეას (trend + სეზონურობა + დღესასწაულები) უფრო მოქნილი ფორმით აკეთებს.

### Prophet (model_experiment_Prophet.ipynb)

Prophet კლასიკური სტატისტიკური მიდგომაა. თითოეული store-dept სერიისთვის ცალკე Prophet-ის მორგება (3,331 მოდელი) ძალიან დიდ დროს მოითხოვდა, ამიტომ top-down მიდგომა ავირჩიეთ:

1. თითო მაღაზიის ჯამურ გაყიდვებზე ვარგებთ Prophet-ს (სულ 45 მოდელი). მოდელს ვუმატებთ yearly seasonality-ს, კონკურსის ოთხ დღესასწაულსა და pre-Christmas კვირას holiday რეგრესორებად.
2. მაღაზიის პროგნოზს დეპარტამენტებზე ვანაწილებთ ისტორიული წილებით. თითო დეპარტამენტის წილი კვირის ნომრის (week-of-year) მიხედვით ითვლება, რომ სეზონური დეპარტამენტები (მაგ. სააღდგომო ან საშობაო საქონელი) სწორად დაიყოს.

შედეგები:

| Fold | WMAE |
|------|------|
| 1 | 54362.5 |
| 2 | 1961.1 |
| 3 | 1756.4 |

პირველი fold-ის შედეგი ძალიან ცუდია, რაც თავად Prophet-ის ბუნებით აიხსნება: ტრენინგის პერიოდი მხოლოდ 35 კვირას მოიცავდა, რაც სრულ წელზე ნაკლებია, ამიტომ yearly seasonality-ის სწავლა შეუძლებელი იყო. ექსტრაპოლაცია კი Thanksgiving/Christmas პერიოდზე მოხვდა. სამაგიეროდ იმ fold-ებზე, სადაც სრული წელი ჰქონდა ნანახი, Prophet-მა ხეზე დაფუძნებულ მოდელებზე უკეთესი შედეგიც კი აჩვენა (1961 და 1756 vs XGBoost-ის 2950 და 2556). აქედან ჩანს, რომ კლასიკურ სტატისტიკურ მოდელებს მინიმუმ ერთი სრული სეზონური ციკლი სჭირდებათ, თუმცა საკმარისი ისტორიის პირობებში ისინი საკმაოდ კონკურენტულ შედეგს აჩვენებს. ტიუნინგის შემდეგ (Prophet_v2, იხ. ზემოთ) პირველი fold-ის პრობლემაც მნიშვნელოვნად შერბილდა.

### DLinear (model_experiment_DLinear.ipynb)

Deep learning-ის მოდელი [DLinear სტატიიდან](https://arxiv.org/abs/2205.13504). იდეა ის არის, რომ სერია moving average-ით იშლება trend და seasonal კომპონენტებად და თითოეულზე ცალკე linear layer მუშაობს. ეს მარტივი არქიტექტურაა, მაგრამ სტატიაში ტრანსფორმერებზე უკეთესი შედეგები აქვს long-term forecasting-ში.

ჩვენი იმპლემენტაცია: 52-კვირიანი window, საიდანაც შემდეგი კვირის პროგნოზი კეთდება. თითო სერია ნორმალიზდება საკუთარი mean/std-ით (წინააღმდეგ შემთხვევაში, დიდი მაღაზიები დომინირებდნენ loss-ში). Loss-ად პირდაპირ weighted MAE გამოვიყენეთ, იმავე 5x წონებით სადღესასწაულო კვირებისთვის, ანუ მოდელი პირდაპირ იმავე მეტრიკაზე ოპტიმიზდება, რომლითაც საბოლოოდ ფასდება. test-ის პერიოდისთვის პროგნოზი autoregressive-ად ეშვება: ყოველი პროგნოზირებული კვირა შემდეგი კვირის window-ში ხვდება.

deep learning-ის ოჯახიდან ჯერ DLinear ავირჩიეთ, სწორედ სტატიის ჰიპოთეზის შესამოწმებლად, შემდეგ ეტაპზე კი შესადარებლად N-BEATS და PatchTST-იც გავტესტეთ, რომ „მარტივი vs ღრმა vs ტრანსფორმერი“ დაპირისპირება ჩვენს მონაცემებზეც გვენახა.

შედეგები (seq_len=52, kernel_size=25, 15 epoch):

| Fold | WMAE |
|------|------|
| 1 | 3789.0 |
| 2 | 3280.8 |
| 3 | 1981.6 |
| საშუალო | 3017.1 |

ორი შედეგი ერთი შეხედვით საეჭვოდ გამოიყურება, თუმცა ორივეს ლოგიკური ახსნა აქვს.

ტრენინგი ძალიან სწრაფია და loss პირველივე ეპოქის შემდეგ სტაბილიზდება (0.674-დან 0.646-მდე). ეს შეცდომა არ არის: მოდელს სულ დაახლოებით 106 პარამეტრი აქვს (ორი 52x1 წრფივი layer) და მყისიერად კონვერგირდება. სწორედ ეს არის DLinear-ის სტატიის მთავარი აზრი, long-term forecasting-ისთვის ძალიან მარტივი მოდელიც საკმარისია.

Full-train WMAE 0-ია, მაგრამ ეს მაჩვენებელი რეალურ ხარისხს არ ასახავს. predict() ნანახი თარიღებისთვის პირდაპირ ისტორიულ მნიშვნელობას აბრუნებს და autoregressive პროგნოზი მხოლოდ უცნობ თარიღებზე ირთვება. CV-ისა და test-ის შეფასებები ამით არ ზიანდება (იქ თარიღები ტრენინგის ისტორიაში არაა), უბრალოდ ამ მოდელისთვის train_wmae როგორც sanity მეტრიკა უსარგებლოა.

### N-BEATS (model_experiment_NBEATS.ipynb)

N-BEATS ([Oreshkin et al. 2019](https://arxiv.org/abs/1905.10437)) DLinear-ის საპირისპირო ფსონია: fully-connected block-ების ღრმა დასტა, სადაც თითო block-ი input window-დან ითვლის backcast-ს (რას ხსნის წარსულში) და forecast-ს, მომდევნო block-ს კი მხოლოდ residual-ი გადაეცემა და პროგნოზები ჯამდება. დაახლოებით 670 ათასი პარამეტრია 106-ის ნაცვლად. დანარჩენი მილსადენი (per-series ნორმალიზაცია, weighted MAE loss, autoregressive პროგნოზი) DLinear-ის იდენტურია, რომ შედარება სუფთა ყოფილიყო.

შედეგები: fold-ები 4245 / 3872 / 2184, holdout 3482. დამატებითმა სიღრმემ ჩვენს მონაცემებზე DLinear-ს ვერ აჯობა, რაც DLinear-ის სტატიის მთავარ დასკვნას ეთანხმება.

### PatchTST (model_experiment_PatchTST.ipynb)

PatchTST ([Nie et al. 2023](https://arxiv.org/abs/2211.14730)) ტრანსფორმერული მიდგომაა: სერია რვაკვირიან patch-ებად იყოფა (როგორც ტექსტი ტოკენებად), patch-ების მიმდევრობას კი transformer encoder ამუშავებს. ეს სწორედ ის არქიტექტურული ხაზია, რომელსაც DLinear-ის სტატია ედავება, ამიტომ საინტერესო იყო საკუთარ მონაცემებზე გვენახა, ვინ იმარჯვებდა.

შედეგები: fold-ები 4378 / 3460 / 2137, holdout 3310. ტრანსფორმერმა ვერც DLinear-ს აჯობა და ვერც ხეზე დაფუძნებულ მოდელებს. ჩვენი ინტერპრეტაცია ისაა, რომ 143 კვირის სიგრძის მონაცემებზე დიდი მოდელის მოქნილობა უფრო ხმაურის მორგებაზე იხარჯება, ვიდრე სიგნალის დაჭერაზე.

### TimesFM (model_experiment_TimesFM.ipynb)

ბონუს ნაწილად Google-ის pretrained foundation მოდელიც გავტესტეთ (timesfm-2.5-200m). ტრენინგი საერთოდ არ სჭირდება: თითო სერიის ისტორიას კონტექსტად ვაწვდით და zero-shot პროგნოზს ვიღებთ. fit მხოლოდ ისტორიას ინახავს, ხოლო predict-ის დროს checkpoint-ი huggingface-დან ჩამოიტვირთება; თვითონ მოდელის წონები pipeline-ის pickle-ში არ ინახება (მთელი pipeline მხოლოდ 4 MB-მდეა).

შედეგები: fold-ები 3491 / 3031 / 1672, holdout 2754. zero-shot მოდელისთვის ეს ძალიან კარგი შედეგია: ყოველგვარი ტრენინგის გარეშე ის ჩვენს პირველი ვერსიის ხე-მოდელებს უტოლდება და deep learning მოდელებს ჯობია კიდეც, თუმცა დატიუნებულ LightGBM_v5-მდე ვერ დადის. ისიც აღსანიშნავია, რომ Prophet-ის მსგავსად, ისტორიის სიგრძეზეა მგრძნობიარე (პირველი fold-ი აქაც ყველაზე ცუდია).

## ლოგირება

- MLflow: ექსპერიმენტები სტრუქტურით `{Model}_Training`, შიგნით run-ები `{Model}_Cleaning`, `{Model}_Feature_Selection` (ნეირონულ მოდელებში `{Model}_Windowing`), `{Model}_Tuning` (სადაც ძებნა ტარდებოდა), `{Model}_CV`, `{Model}_Final`. ლოგები და მოდელის არტეფაქტები თავად რეპოზიტორიაში ინახება (`mlflow.db` და `mlruns/`), ლოკალურად სანახავად: `mlflow ui --backend-store-uri sqlite:///mlflow.db`
- Wandb: ექსპერიმენტების ლოგები გუნდის ორივე წევრის ანგარიშზე ინახება, [gmama23](https://wandb.ai/gmama23-free-university-of-tbilisi-/walmart-sales-forecasting) და [adola23](https://wandb.ai/adola23-free-university-of-tbilisi-/walmart-sales-forecasting)

ექსპერიმენტები Google Colab-ში ეშვება. notebook-ის პირველი cell-ი რეპოს კლონავს, აყენებს დამოკიდებულებებს და Kaggle API-ის საშუალებით იწერს მონაცემებს, ბოლო cell-ი კი განახლებულ ლოგებს (`mlflow.db`, `mlruns/`) რეპოში აბრუნებს push-ით. ნეირონული მოდელებისთვის GPU runtime გამოვიყენეთ.

## შედეგების შედარება

ცხრილში 3-fold CV-ს შედეგები და, სადაც გავზომეთ, ტესტის ფორმის holdout-იც არის მოცემული. სწორედ holdout არის ის მეტრიკა, რომელიც Kaggle-ის ქცევას ყველაზე კარგად იმეორებს.

მთავარი დასკვნები:

საბოლოო გამარჯვებული LightGBM_v5-ია (holdout 1936, Kaggle დაახლოებით 2800). გამარჯვება არა თვითონ არქიტექტურამ, არამედ ნიშნების სწორად აწყობამ მოიტანა: ისტორიული lag-ები, კალენდარული გასწორება და leakage-ის leave-one-out გამოსწორება.

deep learning მოდელებმა ხეზე დაფუძნებულებს ვერ აჯობეს, თანაც მათ შორის ყველაზე მარტივმა DLinear-მა (106 პარამეტრი) აჯობა როგორც ღრმა N-BEATS-ს (670 ათასი პარამეტრი), ისე ტრანსფორმერულ PatchTST-ს. ეს DLinear-ის სტატიის დასკვნის პრაქტიკული დადასტურებაა ჩვენს მასშტაბზე.

Prophet-ი და TimesFM ისტორიის სიგრძეზე ყველაზე მგრძნობიარეა: სრული სეზონური ციკლის გარეშე ორივე ზიანდება, საკმარისი ისტორიით კი ორივე ძალიან კონკურენტულია. TimesFM განსაკუთრებით საინტერესოა, რადგან ამ შედეგს ყოველგვარი ტრენინგის გარეშე აღწევს.

და ყველაზე მნიშვნელოვანი გაკვეთილი: ვალიდაციის დიზაინი მოდელზე მეტად წყვეტს. სანამ სავალიდაციო ფანჯარა ტესტის კალენდარს არ დაემთხვა (სადღესასწაულო პიკებით), ლოკალური შეფასება სისტემატურად გვატყუებდა და leakage-საც ვერ ვამჩნევდით.

## Model Registry და საბოლოო პროგნოზი

model_registry.ipynb ყველა ექსპერიმენტის დასრულების შემდეგ ეშვება და ორ რამეს აკეთებს: ჯერ ყოველი არქიტექტურის საუკეთესო (Final) pipeline-ს არეგისტრირებს Model Registry-ში საკუთარი სახელით (`walmart-{არქიტექტურა}`), რომ ყველა მოდელი Registry-შიც ჩანდეს და სახელით ჩამოტვირთვადი იყოს, შემდეგ კი საერთო საუკეთესოს (ტესტის ფორმის holdout-ის მიხედვით) კანონიკური სახელით `walmart-sales-model` არეგისტრირებს. საბოლოო ჩემპიონი LightGBM_v5-ია.

model_inference.ipynb Registry-დან იტვირთავს მოდელს, პროგნოზს აკეთებს უშუალოდ დაუმუშავებელ test.csv-ზე და ქმნის submission ფაილს, რომელიც Kaggle-ზე იტვირთება. SUBMIT_ARCH პარამეტრით ნებისმიერი დარეგისტრირებული არქიტექტურის გატესტვა შეიძლება, ხოლო ნაგულისხმევად (None) ჩემპიონი იტვირთება.

საბოლოო Kaggle-ის შედეგი (LightGBM_v5): დაახლოებით 2800 WMAE. ლოკალურ holdout-თან (1936) სხვაობა დაახლოებით 900-ია და ეს მოსალოდნელია: 2013 წლის სატესტო პერიოდი ნებისმიერ ლოკალურ ფანჯარაზე რთულია, რადგან მისი ლეიბლები ტრენინგში ვერასდროს მოხვდება. შედარებისთვის, მარტივი „გასული წლის კოპირების“ baseline-იც ანალოგიური ზომით უარესდება ამავე ტესტზე, ანუ ეს სხვაობა მოდელის ნაკლი კი არა, ამოცანის თვისებაა.

## დასკვნა

პროექტში ოთხივე მოთხოვნილი ოჯახის მოდელები გავტესტეთ და შევადარეთ: ხეზე დაფუძნებული (XGBoost, LightGBM და მისი ხუთი ვერსია), კლასიკური სტატისტიკური (ARIMA/SARIMA თეორიულად, Prophet ორ ვერსიად), deep learning (DLinear, N-BEATS, PatchTST) და foundation მოდელი (TimesFM). ყველა არქიტექტურის საუკეთესო ვერსია Pipeline-ად არის შენახული და Model Registry-შია დარეგისტრირებული, საბოლოო პროგნოზი კი Registry-დან ჩატვირთული ჩემპიონით კეთდება.

გზადაგზა სამი მთავარი რამ ვისწავლეთ: ვალიდაციის ფანჯარა ტესტის კალენდარს უნდა ემთხვეოდეს, თორემ ლოკალური შეფასება მატყუარაა; აგრეგატულ ნიშნებში target leakage ადვილად ჩნდება და ლოკალურად კარგ შედეგებს აჩვენებს მაშინაც კი, როცა რეალურად მოდელს აფუჭებს; და დომენური ცოდნა (Christmas-ის კვირის წანაცვლება) ხშირად უფრო მეტს იძლევა, ვიდრე მოდელის გართულება.
