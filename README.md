# Walmart Sales Forecasting

Kaggle-ის [Walmart Recruiting - Store Sales Forecasting](https://www.kaggle.com/competitions/walmart-recruiting-store-sales-forecasting) კონკურსის ამოხსნა. ამოცანაა 45 Walmart-ის მაღაზიის დეპარტამენტების კვირეული გაყიდვების პროგნოზირება. შეფასების მეტრიკა არის WMAE (weighted mean absolute error), სადაც სადღესასწაულო კვირებს 5-ჯერ მეტი წონა აქვს. ამიტომ Super Bowl-ის, Labor Day-ის, Thanksgiving-ისა და Christmas-ის კვირების სწორად პროგნოზირება განსაკუთრებით მნიშვნელოვანია.

## მონაცემები

- train.csv: 421,570 ჩანაწერი. 45 მაღაზია, 81 დეპარტამენტი, 2010-02-05 დან 2012-10-26 მდე
- test.csv: 115,064 ჩანაწერი, 2012-11-02 დან 2013-07-26 მდე
- features.csv: Temperature, Fuel_Price, CPI, Unemployment, MarkDown1-5
- stores.csv: მაღაზიის ტიპი (A/B/C) და ზომა

## EDA (eda.ipynb)

პირველ ეტაპზე მონაცემები დეტალურად გავიარეთ და რამდენიმე მნიშვნელოვანი რამ გამოჩნდა.

MarkDown სვეტები თითქმის ცარიელია: 2010 წელს 100%, 2011 წელს დაახლოებით 85% missing არის, რეალურად მხოლოდ 2012 წლიდან ივსება. მათზე დიდი იმედის დამყარება არ შეიძლება. NA-ები 0-ით შევავსეთ, რადგან მარკდაუნის არარსებობა ფაქტობრივად ნიშნავს რომ აქცია არ იყო.

Weekly_Sales ძლიერ skewed არის. მედიანა 7,612-ია, საშუალო 15,981, მაქსიმუმი 693,099. 1,285 ჩანაწერი უარყოფითია (სავარაუდოდ დაბრუნებები), ისინი 0-ზე დავჭერით.

ყველაზე საინტერესო აღმოჩენა ის იყო, რომ Christmas-ის IsHoliday ალამი არასწორ კვირაზეა. დატასეტში სადღესასწაულოდ 31 დეკემბრის კვირაა მონიშნული, მაგრამ რეალური პიკი წინა კვირაშია: 2010-12-24 კვირის ჯამური გაყიდვები 80.9 მლნ იყო, "ოფიციალური" სადღესასწაულო 2010-12-31 კვირის კი მხოლოდ 40.4 მლნ. ამის გამო ყველა მოდელს დავამატეთ ცალკე IsPreChristmas ფიჩერი.

დეპარტამენტები ერთმანეთისგან ძალიან განსხვავდება. Dept 92-ის საშუალო 75,205-ია, ბოლო დეპარტამენტების კი თითქმის 0. Store და Dept იდენტიფიკატორები ყველაზე ძლიერი სიგნალია.

გარე ფაქტორები სუსტია. Temperature, Fuel_Price, CPI და Unemployment ცალ-ცალკე თითქმის არ კორელირებს გაყიდვებთან.

test-ში 11 (Store, Dept) წყვილი საერთოდ არ არსებობს train-ში. ამან თითო სერიაზე დამოკიდებული მოდელებისთვის (Prophet, DLinear) fallback ლოგიკის დამატება მოითხოვა.

## ვალიდაცია

რადგან ამოცანა time-series არის, ჩვეულებრივი random K-Fold არ გამოდგება, მომავლის მონაცემები წარსულში "გაჟონავდა". ამიტომ ყველა მოდელს ერთნაირი, დროზე დაფუძნებული 3-fold სქემით ვაფასებთ: ყოველ fold-ში ვსწავლობთ მხოლოდ წარსულზე და ვაფასებთ მომდევნო დაახლოებით 8 თვეზე. მეტრიკა ყველგან WMAE-ა, იგივე წონებით რაც kaggle-ზე.

ერთი დეტალი: პირველი fold-ის სატრენინგო პერიოდი მხოლოდ 35 კვირაა, რაც DLinear-ის 52-კვირიან input window-ზე ნაკლებია. ამან თავიდან crash გამოიწვია და მოდელს fallback დავუმატეთ (window ავტომატურად პატარავდება, თუ ისტორია არ ჰყოფნის).

## მოდელები

ყველა მოდელი შენახულია sklearn Pipeline-ად, რომელიც პირდაპირ raw მონაცემებზე ეშვება (preprocessing pipeline-შივეა ჩაშენებული).

### XGBoost (model_experiment_XGBoost.ipynb)

ფიჩერები: Store, Dept, Size, Type, IsHoliday, თარიღის კომპონენტები (Year, Month, Week), IsPreChristmas, MarkDown1-5, ეკონომიკური ინდიკატორები. Feature importance-მა აჩვენა რომ ყველაზე მნიშვნელოვანია Dept (0.25) და Size (0.22). საინტერესოა რომ ჩვენი დამატებული IsPreChristmas მე-6 ადგილზეა (0.06), ანუ Christmas-ის კვირის კორექციამ რეალურად იმუშავა.

შედეგები (n_estimators=500, max_depth=8, lr=0.05):

| Fold | WMAE |
|------|------|
| 1 | 4116.5 |
| 2 | 2950.4 |
| 3 | 2555.8 |
| საშუალო | 3207.6 |

პირველი fold-ის შედარებით ცუდი შედეგი ლოგიკურია. ტრენინგში ჯერ სრული წელიც არ იყო და მოდელს სეზონურობის დანახვა არ შეეძლო. Full-train WMAE 2140-ია.

### LightGBM (model_experiment_LightGBM.ipynb)

იგივე ფიჩერები რაც XGBoost-ს, რომ შედეგები პირდაპირ შედარებადი იყოს. განსხვავება ჰიპერპარამეტრებშია: LightGBM leaf-wise იზრდება და მისთვის num_leaves (127) უფრო მნიშვნელოვანი პარამეტრია ვიდრე max_depth. ტრენინგი საგრძნობლად სწრაფია.

შედეგები (n_estimators=1000, num_leaves=127, lr=0.05):

| Fold | WMAE |
|------|------|
| 1 | 3988.6 |
| 2 | 2845.2 |
| 3 | 2265.8 |
| საშუალო | 3033.2 |

სამივე fold-ზე XGBoost-ზე უკეთესია (საშუალო 3033 vs 3208). Full-train WMAE 1682-ია, ტრი-მოდელებს შორის ყველაზე დაბალი. leaf-wise ზრდა მეტ მოქნილობას აძლევს და CV-ზე ოვერფიტის ნიშნები არ ჩანს.

### ARIMA / SARIMA

კლასიკური ოჯახის საბაზისო მოდელებია, თეორიულად გავარჩიეთ მაგრამ ტრენინგზე დრო არ დაგვიხარჯავს. ARIMA(p,d,q) სამი კომპონენტისგან შედგება: AR(p) ნიშნავს რომ მიმდინარე მნიშვნელობა წარსული p მნიშვნელობის წრფივი კომბინაციაა, MA(q) წარსული q პროგნოზის შეცდომისა, I(d) კი იმას რომ სერია d-ჯერ სხვაობდება (differencing) რათა trend მოშორდეს და სტაციონარული გახდეს. SARIMA ამას სეზონურ (P,D,Q,s) კომპონენტს უმატებს, ჩვენს კვირეულ მონაცემებზე s=52 იქნებოდა.

პრაქტიკაში ვერ გამოვიყენებდით რამდენიმე მიზეზით: 3,331 სერიაზე თითოეულისთვის order-ის შერჩევა პრაქტიკულად შეუძლებელია, s=52 სეზონური პერიოდი SARIMA-სთვის გამოთვლითად ძალიან მძიმეა, და მრავალი დღესასწაულის ეფექტის ჩამატება Prophet-ში ბევრად მარტივია. ამიტომ კლასიკური ოჯახიდან Prophet ავირჩიეთ, რომელიც იმავე იდეას (trend + სეზონურობა + დღესასწაულები) უფრო მოქნილი ფორმით აკეთებს.

### Prophet (model_experiment_Prophet.ipynb)

კლასიკური სტატისტიკური მიდგომა. თითო store-dept სერიაზე ცალკე Prophet-ის მორგება (3,331 მოდელი) ძალიან ნელი იქნებოდა, ამიტომ top-down მიდგომა ავირჩიეთ:

1. თითო მაღაზიის ჯამურ გაყიდვებზე ვარგებთ Prophet-ს (სულ 45 მოდელი). yearly seasonality პლუს კონკურსის 4 დღესასწაული და pre-Christmas კვირა holiday რეგრესორებად.
2. მაღაზიის პროგნოზს დეპარტამენტებზე ვანაწილებთ ისტორიული წილებით. თითო დეპარტამენტის წილი კვირის ნომრის (week-of-year) მიხედვით ითვლება, რომ სეზონური დეპარტამენტები (მაგ. სააღდგომო ან საშობაო საქონელი) სწორად დაიყოს.

შედეგები:

| Fold | WMAE |
|------|------|
| 1 | 54362.5 |
| 2 | 1961.1 |
| 3 | 1756.4 |

პირველი fold-ის შედეგი კატასტროფულია და ეს თავად Prophet-ის ბუნებიდან გამომდინარეობს: ტრენინგში მხოლოდ 35 კვირა იყო, სრულ წელზე ნაკლები, და yearly seasonality-ის სწავლა შეუძლებელი იყო. ექსტრაპოლაცია კი Thanksgiving/Christmas პერიოდზე მოხვდა. სამაგიეროდ იმ fold-ებზე, სადაც სრული წელი ჰქონდა ნანახი, Prophet-მა ტრი-მოდელებზე უკეთესი შედეგიც კი აჩვენა (1961 და 1756 vs XGBoost-ის 2950 და 2556). დასკვნა: კლასიკურ სტატისტიკურ მოდელებს მინიმუმ ერთი სრული სეზონური ციკლი სჭირდებათ, თუმცა საკმარისი ისტორიის პირობებში ძალიან კონკურენტულია.

### DLinear (model_experiment_DLinear.ipynb)

Deep learning მოდელი [DLinear სტატიიდან](https://arxiv.org/abs/2205.13504). იდეა ისაა, რომ სერია moving average-ით იშლება trend და seasonal კომპონენტებად და თითოეულზე ცალკე linear layer მუშაობს. მარტივი არქიტექტურაა, მაგრამ სტატიაში ტრანსფორმერებზე უკეთესი შედეგები აქვს long-term forecasting-ში.

ჩვენი იმპლემენტაცია: 52-კვირიანი window, საიდანაც შემდეგი კვირის პროგნოზი კეთდება. თითო სერია ნორმალიზდება საკუთარი mean/std-ით (სხვანაირად დიდი მაღაზიები დომინირებდნენ loss-ში). Loss-ად პირდაპირ weighted MAE გამოვიყენეთ, იგივე 5x წონებით სადღესასწაულო კვირებზე, ანუ მოდელი პირდაპირ იმ მეტრიკაზე ოპტიმიზდება რომლითაც ფასდება. test-ის პერიოდისთვის პროგნოზი autoregressive-ად ეშვება: ყოველი ნაპროგნოზები კვირა შემდეგი კვირის window-ში ხვდება.

DLinear ავირჩიეთ N-BEATS/TFT/PatchTST-ის ნაცვლად როგორც ჰიპოთეზის ტესტი: თუ სტატია მართალია და მარტივი დეკომპოზიცია ორ წრფივ layer-თან ერთად ტრანსფორმერულ არქიტექტურებს ჯობია, ჩვენს კვირეულ მონაცემებზეც უნდა გამოჩნდეს.

შედეგები (seq_len=52, kernel_size=25, 15 epoch):

| Fold | WMAE |
|------|------|
| 1 | 3789.0 |
| 2 | 3280.8 |
| 3 | 1981.6 |
| საშუალო | 3017.1 |

ორი რამ ერთი შეხედვით საეჭვოდ გამოიყურება, მაგრამ ახსნადია.

ტრენინგი ძალიან სწრაფია და loss პირველივე ეპოქის შემდეგ სტაბილიზდება (0.674 დან 0.646 ზე). ეს ბაგი არაა: მოდელს სულ დაახლოებით 106 პარამეტრი აქვს (ორი 52x1 წრფივი layer) და მყისიერად კონვერგირდება. სწორედ ეს არის DLinear-ის სტატიის მთავარი აზრი, long-term forecasting-ისთვის უმარტივესი მოდელიც საკმარისია.

Full-train WMAE 0-ია, მაგრამ ეს არაფერს ნიშნავს. predict() ნანახი თარიღებისთვის პირდაპირ ისტორიულ მნიშვნელობას აბრუნებს და autoregressive პროგნოზი მხოლოდ უცნობ თარიღებზე ირთვება. CV და test შეფასებები ამით არ ზიანდება (იქ თარიღები ტრენინგის ისტორიაში არაა), უბრალოდ ამ მოდელისთვის train_wmae როგორც sanity მეტრიკა უსარგებლოა.

ექსპერიმენტები Google Colab-ში ეშვება. notebook-ის პირველი cell რეპოს კლონავს, აყენებს დამოკიდებულებებს და kaggle API-თ იწერს მონაცემებს, ბოლო cell კი განახლებულ ლოგებს (`mlflow.db`, `mlruns/`) რეპოში აბრუნებს push-ით.

საუკეთესო საშუალო შედეგი DLinear-ს აქვს, თუმცა LightGBM-თან სხვაობა მინიმალურია (დაახლოებით 0.5%). Prophet-ის საშუალო პირველი fold-ის კატასტროფამ გააფუჭა, საკმარისი ისტორიის მქონე fold-ებზე კი ის საუკეთესოა, ანუ მხოლოდ საშუალო CV შედეგი სრულ სურათს ყოველთვის არ იძლევა. ორივე ტრი-მოდელი სტაბილურია ყველა fold-ზე, deep learning და კლასიკური მოდელები კი ისტორიის სიგრძეზე ბევრად მგრძნობიარეა.

## CV-ის და Kaggle-ის შეუსაბამობა

აქ მთავარ პრობლემას წავაწყდით: ლოკალური CV ერთ ამბავს ამბობდა, Kaggle კი სულ სხვას. ყველა მოდელი ლოკალურად 2000-2700 იღებდა, Kaggle-ზე კი 3200-3800. ცალკე diagnostic.ipynb დავწერეთ ამის გასარკვევად და აი რა აღმოჩნდა:

- submission-ის ფორმატი სწორია, sampleSubmission-თან ზუსტად ემთხვევა, ესე იგი ბაგი submission-ში არ არის.
- ჩვენი 3-fold CV-ის fold 2 ერთადერთია რომელშიც ნოემბერი და დეკემბერი საერთოდ არ არის, ანუ ის ხელოვნურად მარტივია (naive baseline იქ 1832-ს იღებს). სწორედ ის ამცირებდა საშუალო CV-ს.
- ავაწყვეთ holdout რომელიც Kaggle-ის ტესტის კალენდარულ ფორმას იმეორებს: ოქტომბრამდე ვსწავლობთ, შემდეგი ნოემბრიდან ივლისამდე ვაფასებთ. ამ holdout-ზე naive baseline-იც და მოდელებიც ბევრად უფრო რეალურ, მაღალ WMAE-ს იძლევა.

მთავარი დასკვნა: 2012-2013 ტესტის პერიოდი უბრალოდ უფრო რთულია ვიდრე train-ის ნებისმიერი ფანჯარა. naive "შარშანდელი წლის გამეორება" თავადვე დაახლოებით 3200-ს იღებს Kaggle-ზე. ანუ 3200 ცუდი შედეგი არ იყო (708 გუნდიდან დაახლოებით 160-ე ადგილი), მაგრამ მის გასაუმჯობესებლად საჭირო იყო კონკურსზე მორგებული ხერხი და არა უბრალოდ tuning.

## ლაგ ფიჩერები და leakage (v3, v4, v5)

Kaggle-ის გასაუმჯობესებლად ისტორიული ფიჩერები დავამატეთ, მაგრამ გზაზე მნიშვნელოვანი შეცდომა დავუშვით და გამოვასწორეთ.

- v3: დავამატეთ lag-52 (შარშანდელი იმავე კვირის გაყიდვები) და per-series აგრეგატები. ლოკალურად გაუმჯობესდა, Kaggle კი 3400-მდე გაუარესდა.
- v4: დავამატეთ კალენდარზე გასწორებული lag-ები plus cross-store department shape. Christmas-ის კვირის ცდომილების გამო (რაც EDA-ში აღმოვაჩინეთ) ყოველი კვირის 7 დღეს ცალკე ვუყურებთ და შარშანდელ კვირებს დღეების გადაფარვის მიხედვით ვურევთ, ასე რომ სადღესასწაულო პიკის გადანაცვლება თვითონ ფიჩერშია ჩადებული. ლოკალურად კიდევ გაუმჯობესდა (2200), Kaggle კი 3700-მდე გაუარესდა.
- ტენდენცია აშკარა იყო: რაც მეტ ისტორიულ აგრეგატ ფიჩერს ვამატებდით, Kaggle მით უარესდებოდა. ეს target leakage-ის ნიშანია. Series_Mean და Series_WoyMean აგრეგატები fit-ის დროს თვითონ იმ სტრიქონსაც შეიცავდნენ რომელსაც წინასწარმეტყველებდნენ (week-of-year აგრეგატში სტრიქონის საკუთარი გაყიდვა მისი მნიშვნელობის 33-50%-ია). მოდელი სწავლობდა პასუხის ნაწილობრივ წაკითხვას, ტესტზე კი, სადაც სტრიქონი აგრეგატში აღარ არის, ეს დაყრდნობა ქრებოდა.
- v5: leakage გავასწორეთ leave-one-out აგრეგატებით. ყოველ სტრიქონს აგრეგატიდან საკუთარი წვლილი აკლდება ((sum - own) / (n - 1)), ტესტის სტრიქონები კი (რომლებიც fit-ში არ იყო) სრულ აგრეგატს იღებენ. ასე train-ის და test-ის ფიჩერებს ზუსტად ერთი და იგივე მნიშვნელობა აქვთ. ჰიპერპარამეტრებს ორ ტესტ-ფორმის holdout-ის საშუალოზე ვირჩევთ, რომ ერთ ფანჯარაზე მორგება არ მოხდეს.

v5-ის CV fold-ები: 4588.7 / 1973.2 / 1600.9 (fold 3 ყველა მოდელს შორის საუკეთესოა). ტესტ-ფორმის holdout-ზე მან 1936 აჩვენა, რაც naive baseline-ს (2074 იმავე ფანჯარაზე) ჯობია, ესე იგი პირველად გავაჯობეთ "შარშანდელი წლის გამეორებას" იმ ფანჯარაზე რომელიც Kaggle-ს იმეორებს.

## Model Registry და საბოლოო პროგნოზი

model_registry.ipynb ერთჯერადად ეშვება ყველა ექსპერიმენტის შემდეგ, MLflow-დან კითხულობს ყველა არქიტექტურის CV შედეგს (fold 1-2, fold 0-ს გამოვტოვებთ რადგან იქ სრული წლის ისტორია არ არის), ირჩევს საუკეთესოს და მის Final pipeline-ს არეგისტრირებს Model Registry-ში სახელით `walmart-sales-model`. ამჟამად დარეგისტრირებულია LightGBM_v5.

model_inference.ipynb Registry-დან იტვირთავს მოდელს, პროგნოზს აკეთებს უშუალოდ raw test.csv-ზე და აგენერირებს submission-ს. `SUBMIT_ARCH` ცვლადით შესაძლებელია კონკრეტული არქიტექტურის (მაგ. LightGBM_v5) პირდაპირ გაშვება Registry-ის არჩევანის ნაცვლად.

საუკეთესო Kaggle შედეგი: დაახლოებით 2800 (WMAE, LightGBM_v5). ეს პირველი რეალური გაუმჯობესებაა საწყისი 3200-დან და leakage-ის გასწორების პირდაპირი შედეგია.

## შემდეგი ეტაპი

- ensemble: LightGBM_v5, Prophet_v2 და DLinear განსხვავებულ შეცდომებს უშვებენ (tree, დეკომპოზიცია, autoregressive), ამიტომ მათი პროგნოზების შერევა სავარაუდოდ კიდევ დაწევს WMAE-ს.
- Registry-ის კრიტერიუმის CV-დან ტესტ-ფორმის holdout-ზე გადატანა, რომ დარეგისტრირებული მოდელი ნამდვილად საუკეთესო იყოს Kaggle-ისთვის და არა მხოლოდ CV-ზე.
